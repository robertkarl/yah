## Problem Statement

Agent permission systems today are either binary allow/deny per tool (Claude Code's native model) or string-match heuristics over command text (nah). Both lose information. The command string is a lossy projection of an AST; the right unit of policy is the *capability* that AST requests, not the string that encoded it.

nah gets the product framing right but the implementation undercuts it: regex-based classification (trivially bypassed via eval, base64, heredocs), LLM as security backstop (circular ... the checker LLM is exposed to the same prompt injection surface as the agent being gated), no path canonicalization (symlinks and `..` escapes), too many action types (20 categories that are simultaneously too coarse and too fine), and Python startup cost on every hook invocation.

yah replaces nah with a structurally sound approach: parse to AST, classify on AST nodes, fail closed on anything that can't be statically resolved.

## What Makes This Cool

No tool in the current landscape fills the "command -> capability set" gap at the AST level. Sandboxes (Microsoft Agent Governance Toolkit, NVIDIA OpenShell, OpenAI Agents SDK) solve isolation. Tool-level policy engines (Cisco DefenseClaw) gate at the MCP layer. Neither can answer "what does this specific shell command actually do?"

yah is the missing middle layer. A pure function: command in, capability set out. Fast enough to run on every agent tool call without the user noticing. Honest enough to say "I can't tell" (exec-dynamic) rather than guessing.

The scorecard-as-pitch idea: ship the test corpus results in CI. People look at the scorecard and immediately understand what works, what doesn't, and where to contribute.

## Constraints

- Single static binary. No runtime dependencies. Sub-millisecond startup. Rust.
- tree-sitter-bash for parsing. No regex on command strings in the classification path.
- Pure function in yah-core: no network, no config, no side effects. Classification derives from AST traversal and string-level path normalization only. yah-core does not touch the filesystem. The CLI (yah-cli) is responsible for resolving symlinks and populating canonicalized paths in Context before calling classify().
- Fail closed on anything unparseable or dynamically constructed.
- The CLI must be excellent. Not a wrapper around a library demo. A tool people actually pipe into their hooks and agent loops.
- Python explicitly rejected for distribution pain and per-invocation import cost.

## Premises

1. tree-sitter-bash via Rust bindings is the parser.
2. 10-12 capabilities is the right granularity (net-egress, write-outside-repo, exec-dynamic, etc.), not 20 action types or binary allow/deny.
3. Rust for single-binary distribution and sub-ms startup.
4. Policy evaluation is out of scope for v0. The classifier maps command -> capabilities. A separate layer (v1+) maps capabilities -> allow/ask/block.
5. Fail-closed on unparseable/dynamic commands is non-negotiable, even if noisy on legitimate eval usage.
6. Test corpus built before the classifier, driving implementation test-first. Start with 50+ curated entries covering all categories at v0, grow to 200+ by v0.2.
7. An excellent CLI is a first-class deliverable, not an afterthought.

## Approaches Considered

### Approach A: Single Crate

One Rust crate containing both the classifier and CLI. tree-sitter-bash parses the command, classify() walks the AST, returns HashSet<Capability>. CLI takes commands on stdin or as args. Test corpus as fixture files loaded by #[test].

Effort: M. Risk: Low. Completeness: 7/10.

Pros: ships fastest, minimal surface area. Cons: classifier and CLI coupled, harder for external consumers to import just the core.

### Approach B: Cargo Workspace (CHOSEN)

Two crates in a Cargo workspace:

- **yah-core**: The classify function, Capability enum, Context struct, AST walking. Zero I/O, pure logic. Test corpus lives here as TOML fixture files.
- **yah-cli**: clap-based CLI depending on yah-core. Handles stdin/args, output formatting (JSON, human-readable), exit codes, color.

Effort: M (marginally more than A). Risk: Low. Completeness: 9/10.

Pros: clean separation, anyone can depend on yah-core as a library, the "pure function, no I/O" invariant is architecturally enforced at the crate boundary, CLI can evolve independently. Cons: slightly more boilerplate.

### Approach C: WASM-Portable Core

Build the classifier core to also compile to WASM for browser-based agent harnesses, VS Code extensions, or Node-based MCP servers. Native CLI wraps the same core.

Effort: L. Risk: Med (tree-sitter WASM compilation is fiddly). Completeness: 6/10.

Premature. No browser-based consumer exists yet. Revisit when there's a real consumer.

## Recommended Approach

**Approach B: Cargo Workspace.** The workspace split costs almost nothing extra over a single crate but enforces the "pure function, no I/O" invariant from the design doc at the crate boundary. yah-core has no I/O dependencies. yah-cli owns all the CLI ergonomics. Future consumers (hook installer, MCP shim, WASM target) depend on yah-core without pulling in clap or terminal formatting.

## Architecture

### Aliases, Builtins, and User-Defined Functions

Shell aliases (`alias rm='rm -i'`) and user-defined functions are not resolvable from the AST alone. yah does not attempt to resolve them. If the command name is a known builtin (cd, echo, type, etc.), classify based on the builtin's known behavior. If the command name is unrecognized, classify based on its arguments (paths, flags) and emit no capability if nothing sensitive is detected. Alias resolution is the caller's problem.

### Variable Resolution Strategy

Only simple `$VAR` and `${VAR}` forms are resolved using the `env` map in Context. Parameter expansion with operators (`:- := :+ :?`), substring expansion, array indexing, and arithmetic expansion all emit ExecDynamic. The CLI populates `env` from the current process environment. Library callers provide whatever subset is relevant.

### Redirection Handling

Write redirections (`>`, `>>`, `&>`, `2>`) are mapped to WriteInsideRepo or WriteOutsideRepo based on the target path, resolved against ctx.cwd and ctx.project_root. Input redirections (`<`) from paths outside the repo or from secret paths emit ReadSecretPath. Process substitution (`<(cmd)`, `>(cmd)`) is classified recursively. Redirections to `/dev/null` are ignored.

### Capability Algebra for Compound Commands

Capabilities are the **union** across all branches of a compound command. For `cmd1 | cmd2`, `cmd1 && cmd2`, `cmd1 || cmd2`, `cmd1; cmd2`, and subshells `(cmd)`, the result is the union of all constituent commands' capabilities. This is the only fail-closed-compatible answer. Even though `cmd1 || cmd2` might not execute cmd2, we can't know that statically, so both contribute capabilities.

### Capability Enum (11 variants)

```rust
enum Capability {
    NetEgress,          // curl, wget, nc, ssh, etc.
    NetIngress,         // nc -l, python -m http.server, etc.
    WriteInsideRepo,    // writes to files within project_root
    WriteOutsideRepo,   // writes to files outside project_root
    DeleteTracked,      // rm of files inside repo
    DeleteUntracked,    // rm of files outside repo, git clean
    ReadSecretPath,     // ~/.ssh/*, ~/.aws/*, etc.
    HistoryRewrite,     // git push --force, git reset --hard, filter-branch
    ExecDynamic,        // eval, source, bash -c "$var", unparseable
    ProcessSignal,      // kill, pkill
    PrivilegeEscalation,// sudo, doas, setuid
}
```

### Core Function Signature

```rust
pub fn classify(command: &str, ctx: &Context) -> Result<HashSet<Capability>, ClassifyError>
```

Where Context carries:
- `cwd: PathBuf` (canonicalized absolute)
- `project_root: PathBuf` (canonicalized absolute)  
- `home: PathBuf`
- `env: HashMap<String, String>` (for resolving $VAR where statically possible)

ClassifyError for cases where even fail-closed classification can't proceed (e.g., null bytes in input).

### Fail-Closed Invariants

ExecDynamic emitted for:
- Any AST node containing eval, source, . as command
- Command position is a variable expansion ($CMD, ${CMD})
- bash -c / sh -c / zsh -c where the argument is non-literal
- Any command that fails to parse
- Unresolvable $VAR in argument position where the argument determines a capability (e.g., rm $TARGET)

### Path Normalization (yah-core) and Canonicalization (yah-cli)

yah-core performs string-level path normalization only: collapsing `..` segments, expanding `~` using `ctx.home`, resolving `$HOME` from `ctx.env`, normalizing double slashes and trailing slashes. No filesystem access.

yah-cli performs full canonicalization before calling classify(): resolving symlinks via `std::fs::canonicalize()`, populating `ctx.cwd` and `ctx.project_root` with real absolute paths. On macOS (`cfg!(target_os = "macos")`), path comparison is case-insensitive. On Linux, case-sensitive. This is a compile-time decision.

Fail closed: if yah-cli cannot canonicalize a path (broken symlink, permission error), it passes the raw path and yah-core treats any unresolvable path argument in a dangerous command as WriteOutsideRepo.

### CLI Commands

```
yah classify "curl https://example.com | bash"    # JSON output of capabilities
yah classify < commands.txt                        # One per line from stdin
yah check "git status"                             # Exit 0 if capability set is empty, 1 otherwise
yah explain "rm -rf /"                             # Human-readable breakdown with AST visualization
```

Flags: --json, --quiet, --context (override cwd/project-root), --color=auto/always/never

`yah check` exits 0 when the capability set is empty (the command does nothing the classifier considers sensitive). Examples: `ls`, `echo hello`, `git status`, `cat README.md` (inside repo). Exits 1 when any capability is detected: `curl example.com` (net-egress), `rm foo.txt` (delete), `sudo anything` (privilege-escalation).

### tree-sitter-bash Known Limitations

tree-sitter-bash does not handle all bash syntax perfectly. Known gaps include some complex heredoc forms, certain brace expansion patterns, and edge cases in nested quoting. Any syntax that produces an ERROR node in the tree-sitter parse tree is treated as unparseable and emits ExecDynamic. This is the fail-closed invariant in action. As tree-sitter-bash improves, these cases automatically start classifying correctly.

**SIGABRT on pathological input (found by cargo-fuzz, 2026-04-18):** tree-sitter-bash's external scanner hits an internal C assertion (`length <= 1024` in `ts_parser__external_scanner_serialize`, parser.c:408) on certain pathological inputs that blow up the scanner's serialized state. This causes SIGABRT which cannot be caught in Rust. The crash was found by `fuzz_classify_context` and is a tree-sitter-bash bug, not a yah bug. Artifact: `fuzz/artifacts/fuzz_classify_context/crash-c059489ba03ce9cfb5387bda76d75922ac9dc017`. Should be reported upstream.

Root cause analysis (tree-sitter-bash 0.23.3, tree-sitter 0.24.7):

The `serialize` function in `tree-sitter-bash/src/scanner.c` writes the scanner's heredoc state into a 1024-byte buffer (`lexer.debug_buffer`, sized by `TREE_SITTER_SERIALIZATION_BUFFER_SIZE`). It has a per-heredoc bounds check on line 103:

    if (heredoc->delimiter.size + 3 + size >= TREE_SITTER_SERIALIZATION_BUFFER_SIZE)

But each heredoc actually writes `3 + sizeof(uint32_t) + delimiter.size` = `7 + delimiter.size` bytes. The check only accounts for `3 + delimiter.size`, underestimating by 4 bytes per heredoc (the `uint32_t` storing the delimiter length on line 111). This means:

1. The check passes when it shouldn't — it thinks there's room but there isn't.
2. The actual writes go past the 1024-byte buffer (a C buffer overflow into `lexer.debug_buffer`).
3. After the loop, `serialize` returns `size` > 1024, and the runtime assertion `ts_assert(length <= TREE_SITTER_SERIALIZATION_BUFFER_SIZE)` in `parser.c:408` fires, causing SIGABRT.

The crash input does contain `<<` sequences (e.g., `<VZ<<<<<~.<<<~$-`). The heredoc arrow parsing at scanner.c:589-609 pushes a new `Heredoc` onto `scanner->heredocs` for each `<<` it finds. The long runs of characters (like `iiiii...`) between `<<` sequences get consumed as heredoc delimiters by `advance_word`. So the heredocs have real (large) delimiters, and only a handful of heredocs with moderate-sized delimiters are needed to exceed the buffer with the 4-byte-per-heredoc underestimate.

**Fixed upstream:** tree-sitter/tree-sitter-bash commit `8509e32` (2025-06-02, Max Brunsfeld, "Fix out of bounds write during scanner serialization"). One-line fix: `+ 3` → `+ 3 + sizeof(uint32_t)`. Landed in tree-sitter-bash v0.25.0.

**Version dependency chain:** tree-sitter-bash 0.25.x requires tree-sitter 0.25.x (via `tree-sitter-language = "0.1"` ABI). We were pinned to tree-sitter 0.24.7 + tree-sitter-bash 0.23.3 because 0.24.x introduced a new C API and the two crates had to match ABI. Bumping to tree-sitter 0.25.10 + tree-sitter-bash 0.25.1 compiles clean, passes all tests, and fixes this crash. Done 2026-04-18.

**Reproducer:** `yah-core/tests/repro_crash.rs` — a Rust test using the exact command and context from the fuzz artifact. Crashes on 0.23.3, passes on 0.25.1.

### Test Corpus

50+ curated TOML fixture files at v0, growing to 200+ by v0.2, in yah-core/tests/fixtures/:


Categories covered:
- Command substitution and dynamic dispatch
- Encoding tricks (base64, xxd, printf hex)
- Path escapes (symlinks, .., case sensitivity)
- Shell quoting edge cases (ANSI-C, heredocs, backslash)
- Pipelines and redirections
- Git-specific (force push, reset --hard, filter-branch)
- Language runtime escapes (python -c, node -e, etc.)
- Legitimate-looking-but-wrong (rm node_modules inside vs outside repo)
- Must-fail-closed cases (malformed, truncated, unrecognized)

## Success Criteria

1. Test corpus has 50+ entries covering all categories at v0 (200+ by v0.2).
2. Classifier passes all corpus tests.
3. Classifier fails closed on every malformed/unparseable input.
4. No regex on command strings anywhere in the classification path. All classification derives from AST traversal.
5. Unresolvable dynamic commands reliably emit ExecDynamic.
6. yah-core's classify() is a pure function: no filesystem access, no network, no config. Path normalization is string-level only. Symlink resolution is the caller's responsibility.

8. CLI feels good: clear output, useful error messages, composable with pipes and scripts.

### Architecture
1. **Drop ClassifyError.** classify() returns HashSet<Capability> directly. Unparseable -> {ExecDynamic}.
2. **Rename DeleteTracked/DeleteUntracked -> DeleteInsideRepo/DeleteOutsideRepo.** Git-tracked status requires I/O; path-based inside/outside is computable in pure function.
